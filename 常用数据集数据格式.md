# DIP-IMU数据集：以.pkl格式存储；

## 一、.pkl格式

`.pkl` 文件是 Python 中 **Pickle（泡菜）** 模块生成的序列化文件格式。它的全称也可以是 `.pickle`，两者本质上是同一种东西。

**`.pkl` 文件就是把 Python 里的各种对象（如列表、字典、函数、甚至是训练好的机器学习模型）打包成一个二进制文件保存到硬盘上，下次用时再原封不动地还原出来。**

### 1. 核心概念：序列化与反序列化

- **序列化**：把内存中的 Python 对象转化为字节流，存入 `.pkl` 文件。
- **反序列化**：从 `.pkl` 文件中读取字节流，在内存中重新构建成原来的 Python 对象。

### 2. 为什么需要 .pkl 文件？（常见使用场景）

- **保存机器学习/深度学习模型**：这是最常见的场景。训练一个模型可能需要几个小时甚至几天，用 pickle 保存后，下次可以直接加载模型进行预测，无需重新训练。
- **缓存中间数据**：在数据清洗时，处理几 GB 的原始数据可能需要很久。你可以把清洗后的 DataFrame 保存为 `.pkl` 文件，下次直接读取，省去重复计算的时间。
- **保存复杂的数据结构**：JSON 只能保存字典、列表等基础类型，而 `.pkl` 可以保存 Python 的类实例、自定义对象、函数等。



## 二、DIP-IMU数据格式

key数量: 6 

keys: ['gt', 'imu_acc', 'sop', 'sip', 'imu', 'imu_ori'] 

各字段信息: 

gt: type=ndarray, shape=(13778, 72) 

imu_acc: type=ndarray, shape=(13778, 17, 3) 

sop: type=ndarray, shape=(13778, 72) 

sip: type=ndarray, shape=(13778, 72) 

imu: type=ndarray, shape=(13778, 17, 12) 

imu_ori: type=ndarray, shape=(13778, 17, 3, 3)



DIP-IMU 没有给出一个可追踪的绝对空间原点；做人体建模时通常等效为“骨盆根节点为原点（或平移置零）



三、每个字段的解释

1. gt
   含义：人体姿态真值（更准确说是高精度伪真值）。
   格式：形状通常是 (N, 72)。
   解释：72 = 24 个 SMPL 关节 × 每关节 3 维轴角。
   来源：由 SIP 方法在 17 个 IMU 全量数据上离线优化得到，作为训练/评估标签。
2. imu_acc
   含义：IMU 线加速度。
   格式：通常是 (N, 17, 3)。
   解释：N 帧，17 个传感器，每个传感器 3 轴加速度。
3. imu_ori
   含义：IMU 朝向（旋转矩阵）。
   格式：通常是 (N, 17, 3, 3)。
   解释：每帧每个 IMU 一个 3×3 旋转矩阵，表示传感器姿态。
4. imu
   含义：为网络输入预先拼好的 IMU 特征。
   格式：通常是 (N, 17, 12)。
   解释：12 维一般是 3 维加速度 + 9 维旋转矩阵展平（3×3 拉直）。
5. sop
   含义：SOP（Sparse Orientation Poser）基线方法输出的姿态序列。
   格式：通常是 (N, 72)。
   解释：它是一个“姿态估计结果/基线预测”，不是原始传感器量。
6. sip
   含义：SIP（Sparse Inertial Poser）基线方法输出的姿态序列。
   格式：通常是 (N, 72)。
   解释：同样是“姿态估计结果/基线预测”，不是原始 IMU 信号。



imu 是按展平后的 imu_ori + imu_acc 这个顺序拼接的；也就是前 9 维是 imu_ori 展平后得到的 9 维旋转矩阵，后 3 维才是 imu_acc，

imu 的前 9 维等于 imu_ori.reshape(N, 17, 9)，也就是把每个 3×3 旋转矩阵按行依次排成 9 维向量。

imu_acc不包含重力加速度

`imu_ori` 更准确地说是每个 IMU 传感器在当前时刻的朝向旋转矩阵

imu_acc 是在 IMU 自己的局部坐标系，也就是载体坐标系下表示的三维加速度；imu_ori 则给出了这个 IMU 坐标系相对于世界坐标系的朝向。因此，想把 imu_acc 变到世界坐标系下，确实可以利用 imu_ori 做旋转变换。

只是要注意一个细节：到底是用 R·a 还是 R^T·a，取决于 imu_ori 在数据里定义的是“局部到世界”还是“世界到局部”的旋转。也就是说，严谨的说法应该是：imu_acc 是局部系下的加速度，imu_ori 提供了局部系和世界系之间的旋转关系，所以可以借助 imu_ori 把它换算到世界系。





### 根绝对姿态：

就是第 0 个 IMU 在当前统一全局坐标系里的朝向。你可以把它记成：Rroot，它回答的问题是：“此时此刻，整个人体根部传感器在世界坐标系里朝向哪里？”

比如：人整体朝北、朝东、转身了多少；骨盆这一块整体在世界里的旋转状态



真正进入主干网络前，除了根 IMU 以外，其余 5 个 IMU 的姿态都会被转成“相对于根 IMU”的相对姿态。



1. imu_acc: [B, T, 6, 3]

2. imu_ori: [B, T, 6, 3, 3]

   其中：

   1. B 是 batch size
   2. T 是时间长度
   3. 6 是 6 个 IMU
   4. imu_acc 的最后一维 3 是三轴加速度
   5. imu_ori 的最后两维 3×3 是姿态旋转矩阵

这些数据在预处理阶段已经先被统一到了项目自己的全局坐标系里





- 朝向部分：
  - 根传感器保留自己的全局朝向
  - 其他 5 个传感器改成“相对根传感器的朝向”
- 加速度部分：
  - 全部都用根传感器作为参考来重表达
  - 其中非根传感器先减去根传感器加速度，再转到根参考系





网络里的 acc 处理流程是：先保留根 IMU 的原始加速度，并把其余 5 个 IMU 改成相对根的加速度差；然后把这 6 个加速度向量整体统一到根 IMU 参考系下表达。



设 6 个 IMU 的原始加速度分别是：

a0,a1,a2,a3,a4,a5

其中 a0 是根 IMU。
